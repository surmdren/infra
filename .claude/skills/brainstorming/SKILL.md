# Brainstorming — 想法到设计

通过自然的协作对话，将粗糙想法转化为完整的设计方案和规格说明。

先理解当前项目上下文，然后逐一提问来细化想法。理解清楚要构建什么之后，呈现设计方案并获取用户确认。

<HARD-GATE>
在呈现设计方案并获得用户明确批准之前，**不得**调用任何实现 skill、编写任何代码、搭建任何项目结构，或采取任何实现行动。无论项目看起来多简单，此规则一律适用。
</HARD-GATE>

## 反模式："这太简单了，不需要设计"

每个项目都要经历这个流程。待办列表、单函数工具、配置变更——全部如此。"简单"项目恰恰是未经审视的假设造成最多无效工作的地方。设计可以很短（真正简单的项目几句话就够），但必须呈现并获得确认。

## 执行步骤（按序完成）

1. **探索项目上下文** — 检查相关文件、文档、近期提交
2. **提供视觉辅助（可选）** — 若涉及视觉问题，询问是否需要提供线框图/示意图
3. **逐一提问** — 每次只问一个问题，理解目的、约束条件、成功标准
4. **提出 2-3 个方案** — 附权衡分析和推荐意见
5. **分段呈现设计** — 按复杂度分段展示，每段获取用户确认
6. **输出设计文档** — 保存到 `docs/specs/YYYY-MM-DD-<topic>-design.md` 并 commit
7. **进入实现阶段** — 确认后调用对应的实现 skill（如 `/dev-executor`、`/feature-dev`）

## 提问原则

- 每次只问一个问题，等待回答后再问下一个
- 先问最关键的约束（用户/场景/成功标准），再问技术细节
- 如果用户已经提供了足够信息，减少提问，直接呈现方案

## 方案呈现格式

```
## 方案 A：[名称]
[简短描述]
优点：...
缺点：...

## 方案 B：[名称]
...

## 推荐：方案 X
原因：...
```

## 注意事项

- 设计复杂度要和项目复杂度匹配，不要过度设计
- YAGNI 原则：去掉所有"以后可能用到"的功能
- 设计文档是后续开发的输入，写清楚可以减少实现阶段的来回
