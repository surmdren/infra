# Release QA - 全面技术验收

## 定位

**前提**：dev-executor / dev-autopilot 已完成所有模块开发。
**目标**：对照设计文档，逐条验证实现的正确性，在部署前发现所有问题。

与其他测试 Skill 的区别：

| Skill | 驱动源 | 数量 | 时机 |
|-------|--------|------|------|
| `e2e-test-generator` | 用户流程 | 1-2 条 UI 测试 | 开发中 |
| `dev-integration` | DevPlan 模块 | 中等，逐模块 | 开发中 |
| **release-qa** | PRD + Architecture + TechSolution | 全量覆盖 | 发布前 |

## 工作流程

### Step 1: 读取基准文档，构建验收矩阵

并行读取以下文档，提取验收条目：

```
PRD/ 或 *.md (需求文档)
  → 提取：功能需求列表、业务流程、边界条件

Architecture/ (架构文档)
  → 提取：组件职责、组件间接口契约、数据流路径

TechSolution/ (技术方案)
  → 提取：API 端点列表、请求/响应 Schema、数据模型、错误码

DevPlan/checklist.md
  → 提取：已完成模块列表，确认开发范围
```

构建**验收矩阵**（见 `references/verification-matrix.md` 模板），每行是一个验收条目，包含：
- 来源文档
- 验收内容描述
- 测试类型
- 通过标准

### Step 2: 按类型执行验证

#### 2.1 功能完整性验证（PRD 驱动）

对照 PRD 的每个功能点，验证是否已实现且行为正确：

```
对每个 PRD 功能需求：
  1. 找到对应的 API 端点或前端页面
  2. 构造覆盖正常路径 + 边界条件的测试用例
  3. 执行测试，验证响应符合 PRD 描述
  4. 标记：✅ 通过 / ❌ 缺失 / ⚠️ 行为偏差
```

**测试重点**：
- 每个 PRD 功能至少有 1 个 Happy Path 用例
- 关键业务规则（防重复、限流、权限）必须有专用用例
- 边界条件：空值、最大值、特殊字符

#### 2.2 技术正确性验证（Architecture 驱动）

对照架构文档，验证组件行为符合设计：

```
对每个架构组件及其接口契约：
  1. 验证 API 响应结构与 TechSolution Schema 一致
  2. 验证模块间调用链路正确（如 Controller → Service → Repository）
  3. 验证错误处理符合架构设计（错误码、错误格式）
  4. 验证认证/授权机制工作正常
```

**API 契约验证**（调用真实接口，禁止 Mock）：
```
对 TechSolution 中每个 API 端点：
  - 验证 HTTP 方法、路径正确
  - 验证请求参数校验（缺字段→400，无权限→401/403）
  - 验证响应 Schema 与文档一致（字段名、类型、结构）
  - 验证错误码与文档一致
```

#### 2.3 数据流转验证（端到端数据链路）

追踪数据从输入到存储到输出的完整路径：

```
对每条关键数据流（来自 Architecture 数据流图）：
  1. 在入口写入数据（POST / 表单提交 / 事件触发）
  2. 验证中间状态（缓存、消息队列、中间计算结果）
  3. 验证最终存储（数据库记录字段与预期一致）
  4. 通过查询接口取回数据，验证与写入一致
  5. 若有异步处理，等待后验证最终状态
```

典型数据流示例：
```
用户提交表单 → API 接收 → 校验 → 业务处理 → 持久化 → 事件/通知
     ↓              ↓          ↓         ↓           ↓          ↓
  验证请求体    验证入库前  验证规则  验证业务逻辑  验证DB记录  验证副作用
```

#### 2.4 i18n 国际化验证（默认必须执行）

前端默认支持 zh/en 双语（见 TechSolution i18n 规范），以下验证必须通过：

**URL 路由结构**：
```
✅ GET /zh → 返回中文页面（200）
✅ GET /en → 返回英文页面（200）
✅ GET / → 重定向到 /zh 或 /en（302，基于 geo/cookie）
```

**Geo 自动重定向**：
```
✅ 模拟 CN IP 访问 / → 重定向到 /zh
✅ 模拟 US IP 访问 / → 重定向到 /en
✅ middleware.ts 中 request.geo?.country 逻辑正确
```

**Cookie 语言偏好**：
```
✅ 切换语言后 cookie preferred-locale 被正确设置
✅ 携带 cookie preferred-locale=zh 访问 / → 重定向到 /zh（忽略 geo）
✅ 携带 cookie preferred-locale=en 访问 / → 重定向到 /en（忽略 geo）
```

**文案完整性**：
```
✅ /zh 路由下所有页面无英文硬编码文字残留
✅ /en 路由下所有页面无中文硬编码文字残留
✅ 错误提示、Toast、空状态文案均已国际化
```

### Step 3: 执行测试并自动修复

```
while 有失败用例:
  1. 分析失败原因（日志 + 代码 diff 对比文档）
  2. 判断类型：
     - 实现 bug → 修复代码
     - 测试用例理解偏差 → 修正用例
     - 文档与实现不一致 → 记录 discrepancy，询问用户
  3. 修复后重新运行该用例
  4. 确认通过后继续下一个
```

**禁止**：
- 使用 Mock 绕过真实调用
- 修改测试标准以适应错误的实现
- 跳过失败用例继续

### Step 4: 生成验收报告

输出到 `QA/release-qa-report.md`，包含：

```markdown
# Release QA 验收报告

## 执行概览
| 维度 | 总数 | 通过 | 失败 | 偏差 |
|------|------|------|------|------|
| PRD 功能完整性 | N | N | N | N |
| API 契约正确性 | N | N | N | N |
| 数据流转验证 | N | N | N | N |
| i18n 国际化验证 | N | N | N | N |

## 验收矩阵详情
[每条条目的测试结果]

## 问题清单
### ❌ 阻断问题（必须修复才能发布）
### ⚠️ 偏差问题（实现与文档不一致，需确认）

## 修复记录
[已修复的问题及修复方案]

## 结论
- [ ] 所有 PRD 功能已实现且行为正确
- [ ] 所有 API 契约与 TechSolution 一致
- [ ] 所有数据流转正确
- [ ] i18n 路由、Geo 重定向、cookie 偏好、文案完整性全部通过
- [ ] 无阻断问题
- [ ] ✅ 可以进行 dev-deploy 部署
```

## 注意事项

1. **文档优先**：以 PRD/Architecture/TechSolution 为真相来源，实现与文档不符时不轻易调整文档
2. **全量覆盖**：不跳过任何 PRD 功能点，即使开发声称"已完成"
3. **真实调用**：所有测试调用真实运行的系统，需要确保测试环境服务已启动
4. **数据隔离**：使用独立测试数据，测试后清理，不污染开发/生产数据
5. **阻断原则**：任何 ❌ 问题修复前不进入 dev-deploy
