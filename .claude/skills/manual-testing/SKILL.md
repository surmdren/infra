# 人工测试用例生成器

## Overview

读取 PRD（及可选的 Design/pages、TechSolution），为每个功能模块生成可直接交给测试人员执行的测试用例文档。

**输出目录**：

```
ManualTesting/
├── test-plan.md              # 测试计划总览（范围、优先级、环境、入口）
└── test-cases/
    ├── 01-{模块名}.md        # 每个功能模块一个文件
    ├── 02-{模块名}.md
    └── ...
```

---

## 第一步：读取输入文档

按以下优先级读取：

1. `PRD/requirements.md` — 功能需求、用户故事、业务规则（必读）
2. `Design/pages/` — 页面设计规格，补充 UI 交互细节（可选）
3. `TechSolution/` — 了解 API、数据流，用于接口联动测试（可选）

若用户直接提供需求描述（无文件），以描述内容为输入。

---

## 第二步：生成 test-plan.md

**路径**：`ManualTesting/test-plan.md`

必须包含：

| 章节 | 内容 |
|------|------|
| 测试范围 | 列出所有待测模块，注明是否纳入本次测试 |
| 优先级说明 | P0 = 阻断上线 / P1 = 核心功能 / P2 = 次要功能 |
| 测试环境 | 浏览器/设备要求、测试账号、测试数据准备 |
| 测试入口 | 产品 URL 或启动方式 |
| 模块索引 | 链接到各模块测试用例文件 |
| 上线放行标准 | P0 全部通过；P1 通过率 ≥ 95%；P2 缺陷记录但不阻断 |

---

## 第三步：为每个功能模块生成测试用例文件

**路径**：`ManualTesting/test-cases/NN-{模块名}.md`

### 用例格式

每个测试用例遵循以下结构：

```markdown
### TC-{模块缩写}-{序号}：{测试名称}

**优先级**：P0 / P1 / P2
**前置条件**：（用户已登录 / 购物车有商品 / 等）

**测试步骤**：
1. 打开 {页面}
2. 点击 {按钮/链接}
3. 输入 {内容}
4. ...

**预期结果**：
- {具体可观测的结果，避免模糊描述}

**测试结果**：☐ 通过  ☐ 失败  ☐ 跳过

**备注**：（可选，记录实际结果或 Bug 描述）
```

### 必须覆盖的测试类型

每个模块至少覆盖：

1. **Happy Path** — 标准用户流程（P0/P1）
2. **错误处理** — 表单校验、必填项、格式错误（P1）
3. **边界条件** — 空值、最大长度、极端数值（P1）
4. **权限控制** — 未登录访问受保护页面、角色权限（P0）
5. **数据持久化** — 操作后刷新页面数据是否保留（P1）
6. **关键业务联动** — 跨模块数据是否正确传递（P0/P1）

### 通用兼容性检查（在 test-plan.md 末尾附加）

```markdown
## 兼容性检查清单

### 浏览器兼容
- [ ] Chrome（最新版）
- [ ] Safari（最新版）
- [ ] Edge（最新版）
- [ ] Firefox（最新版）

### 响应式布局
- [ ] 桌面端（1440px）：布局完整，无溢出
- [ ] 平板端（768px）：布局适配，操作可用
- [ ] 移动端（375px）：核心功能可用，字体可读

### 性能感知
- [ ] 页面首次加载 < 3 秒（正常网络）
- [ ] 列表页滚动流畅，无卡顿
- [ ] 表单提交后响应 < 2 秒
```

---

## 第四步：输出质量要求

- **步骤必须具体可操作**：「点击右上角蓝色『提交』按钮」而非「点击提交」
- **预期结果必须可观测**：「显示绿色成功提示：订单已创建，编号 #XXXX」而非「提交成功」
- **P0 用例必须详尽**：每个阻断性流程至少 3-5 个用例
- **用例 ID 全局唯一**：`TC-AUTH-001`、`TC-ORDER-001`、`TC-PAY-001`...
