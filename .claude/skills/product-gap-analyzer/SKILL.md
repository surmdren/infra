# Product Gap Analyzer

扫描现有项目代码，推断行业定位和已有功能，结合 Web 搜索的行业最佳实践，识别高价值的功能缺口，输出优先级排序的需求建议。

## 输出

```
docs/
└── ProductGap.md     # 完整分析报告（含优先级需求清单）
```

---

## Phase 0：扫描代码库，推断现有功能

目标是弄清楚"这个产品现在能做什么"，尽量从代码自动推断，减少问用户。

```bash
# 读取项目基本信息
cat README.md 2>/dev/null | head -100
cat package.json 2>/dev/null

# 扫描路由，推断功能模块
find . -type f -name "*.ts" -o -name "*.tsx" -o -name "*.py" | \
  grep -E "route|page|controller|view|api" | head -40

# 扫描页面/视图文件，推断 UI 功能
find . -path "*/pages/*" -o -path "*/app/*" -o -path "*/views/*" | \
  grep -E "\.(tsx|jsx|vue|py)$" | head -40

# 扫描数据模型，推断数据结构和业务领域
find . -name "*.prisma" -o -name "models.py" -o -name "schema.sql" | \
  xargs cat 2>/dev/null | head -100

# 扫描 API 端点
grep -r "router\.\|app\.get\|app\.post\|@Get\|@Post\|createRoute" \
  --include="*.ts" --include="*.py" --include="*.go" -l | head -20
```

整理发现的信息，形成初步的"已有功能清单"。

## Phase 1：确认行业定位

基于扫描结果，推断行业（如：SaaS 项目管理工具 / B2B 电商平台 / 医疗预约系统）并展示给用户：

```
我分析了你的项目，判断这是一个：[行业定位描述]

已识别的核心功能模块：
- [功能1]
- [功能2]
- ...

如果判断有误，请告诉我实际的行业和产品定位。
```

确认后继续（如果用户没有异议，直接继续，不要等太久）。

## Phase 2：行业最佳实践研究

用 WebSearch 搜索该行业的标准功能集和竞品对比，重点关注：

**搜索策略：**
```
搜索1："{行业} SaaS 必备功能 best practices 2024"
搜索2："{竞品名称} features list"（选 2-3 个头部竞品）
搜索3："{行业} product checklist must-have features"
搜索4："{行业} UX best practices"（UX 维度）
搜索5："{行业} security compliance requirements"（安全维度）
```

从搜索结果中提取：
- 行业标准功能集（80% 的同类产品都有的）
- 差异化功能（头部产品特有的增长点）
- UX 最佳实践（常见的体验优化点）
- 安全合规要求（行业标准的安全功能）
- 性能基准（行业标准的性能期望）

## Phase 3：缺口分析

对比"已有功能"和"行业标准"，识别缺口。分四个维度：

**功能维度**：行业标准功能中，项目还没有的
**UX 维度**：用户体验方面的缺失（onboarding、空状态、错误处理、移动端等）
**安全维度**：安全功能缺口（2FA、审计日志、数据加密、权限控制等）
**性能维度**：性能优化缺口（缓存、分页、搜索、CDN 等）

## Phase 4：优先级评分

对每个缺口按以下维度评分（1-5分），综合得出优先级：

| 维度 | 权重 | 说明 |
|------|------|------|
| 用户价值 | 40% | 有多少用户需要？影响核心流程吗？ |
| 业务影响 | 30% | 影响留存/转化/收入吗？ |
| 实现成本 | 20% | 开发复杂度（反向评分，越简单越高） |
| 行业必要性 | 10% | 没有这个功能会被用户认为不专业吗？ |

**优先级定义：**
- **P0**（立即做）：综合分 ≥ 4.0，且影响核心用户路径
- **P1**（下个迭代）：综合分 3.0-3.9，提升产品竞争力
- **P2**（未来规划）：综合分 < 3.0，或实现成本过高

## 注意事项

- **不要过度分析**：缺口分析的目的是帮助决策，不是穷举所有可能的改进。P0 建议控制在 3-5 个，P1 控制在 5-8 个。
- **基于证据**：每个缺口建议都应该有来源（竞品截图/行业报告/用户常见痛点），不要凭空臆测。
- **考虑团队规模**：如果能感知到项目规模较小，P1/P2 的建议要更保守，聚焦高价值低成本的功能。
- **报告生成后**：告知用户可以用 `/requirement-detail` 将 P0 功能转化为完整 PRD，再用 `/feature-dev` 实现。
