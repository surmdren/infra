# 验收矩阵模板

验收矩阵是 release-qa 的核心工作产物，在 Step 1 读取文档后构建，贯穿整个测试过程。

## 矩阵格式

```markdown
# 验收矩阵 - {项目名称}

生成时间：{YYYY-MM-DD}
基准文档：PRD/{file}.md | Architecture/{dir}/ | TechSolution/{dir}/

---
```

## 构建矩阵的方法

### 从 PRD 提取 A 类条目

扫描 PRD 中每个功能模块，为以下内容创建条目：
- 明确的功能动词句（"用户可以..."，"系统应该..."，"当...时..."）
- 业务规则（防重复、限流、权限、状态机转换）
- 数据格式要求（字段必填、格式校验）

**忽略**：UI 展示描述、颜色字体等纯前端视觉要求（非技术层面）

### 从 TechSolution 提取 B 类条目

对每个 API 端点，至少创建：
- 1 个 Schema 验证（请求结构）
- 1 个 Schema 验证（响应结构）
- 1 个 鉴权失败用例（401/403）
- N 个 参数校验用例（缺失必填字段、格式错误）

### 从 Architecture 数据流图提取 C 类条目

对每条有数据变换的流程，验证：
1. 入口数据 = 预期格式
2. 中间转换 = 正确（加密、编号生成、状态变更）
3. 最终存储 = 与输入一致（无字段丢失、无截断）
4. 副作用触发 = 已执行（邮件发送、消息推送、缓存更新）

### 从 Architecture 组件图提取 D 类条目

对每个有特殊职责的中间件/组件：
- 鉴权：有 token / 无 token / 过期 token / 权限不足
- 限流：正常请求 / 超阈值请求
- 错误处理：各类异常的响应格式
- 安全：注入尝试 / XSS 尝试

## 优先级规则

| 优先级 | 条件 | 处理 |
|--------|------|------|
| P0（阻断）| PRD 核心功能缺失 / 数据丢失 / 安全漏洞 | 必须修复才能发布 |
| P1（高）| API 契约不一致 / 业务规则错误 | 强烈建议修复 |
| P2（中）| 边界条件错误 / 错误码偏差 | 记录，本次可接受 |
| P3（低）| 文档描述模糊导致的理解偏差 | 更新文档 |
